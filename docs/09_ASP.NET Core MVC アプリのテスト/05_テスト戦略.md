# テスト戦略

テスト戦略では、何をどの層で守るかを決めます。

| 守りたいこと | 適したテスト |
| --- | --- |
| 業務ルール | 単体テスト |
| DB mapping / query | 統合テスト |
| 認証認可 | 機能テスト |
| 重要な画面操作 | E2E |
| CI での回帰検知 | 単体 + 主要機能テスト |

テストは多ければよいわけではありません。壊れやすく、遅く、意図が読めないテストは開発を遅くします。

重要なのは、変更が多い部分と事故の影響が大きい部分を優先することです。認可、支払い、在庫、権限、金額計算、データ削除などは厚く守る価値があります。

CI では、速いテストを毎回実行し、重い E2E や本番相当 DB テストはタイミングを分けることもあります。
