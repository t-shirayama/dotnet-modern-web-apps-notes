# SPA が向く場面

SPA は、ブラウザー側のアプリケーションが画面状態と操作を管理し、サーバーとは Web API で通信する構成です。

次の条件では SPA が有力です。

- 画面上の操作が多く、ページ全体の再読み込みを避けたい。
- 複雑なクライアント状態を扱う。
- 既に外部クライアント向け Web API が必要である。
- チームが TypeScript、React、Angular、Vue、Blazor WebAssembly などに慣れている。
- フロントエンドとバックエンドを独立して開発、デプロイしたい。

SPA の価値は、ユーザー体験をアプリケーションのように作れることです。ただし、複雑さも増えます。認証トークン、CORS、CSRF、ルーティング、状態管理、bundle size、SEO、アクセシビリティ、エラー通知など、サーバー側アプリとは別の設計論点が必要です。

| 項目 | 従来型 | SPA |
| --- | --- | --- |
| 画面生成 | サーバー | ブラウザー |
| 主な通信 | HTML | JSON API |
| 初期複雑度 | 低め | 高め |
| リッチ UI | 部分対応 | 得意 |
| SEO | 得意 | 対策が必要 |

SPA は強力ですが、すべての Web アプリを SPA にする必要はありません。機能単位で導入するハイブリッド構成も実務的です。
