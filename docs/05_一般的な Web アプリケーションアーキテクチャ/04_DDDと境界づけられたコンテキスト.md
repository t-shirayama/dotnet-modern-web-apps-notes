# DDD と境界づけられたコンテキスト

DDD は、複雑な業務領域をモデル化するための考え方です。すべてのアプリに必要なものではありませんが、業務ルールが増えるほど価値が出ます。

境界づけられたコンテキストは、モデルが有効な範囲を明確にします。同じ「顧客」でも、販売、請求、サポート、マーケティングでは意味や必要な属性が違うことがあります。

```mermaid
flowchart LR
    Sales["販売コンテキスト\nCustomer"] --- Billing["請求コンテキスト\nCustomer"]
    Billing --- Support["サポートコンテキスト\nCustomer"]
```

単一の巨大な共通モデルに寄せると、一部の変更が全体に影響します。境界を分けると重複は発生しますが、変更の独立性が高まります。

ASP.NET Core アプリで DDD を使う場合、Domain model に不変条件を持たせ、Application Service がユースケースを調整し、Infrastructure が永続化を担当する形にすると整理しやすくなります。
