# アーキテクチャ選択の判断

アーキテクチャは将来の変更に備えるものですが、備えすぎると今の開発速度を落とします。

| 条件 | 推奨しやすい構成 |
| --- | --- |
| 学習用、短期、小規模 CRUD | 単一 Web プロジェクト |
| 中規模、責務を分けたい | Web + Application + Infrastructure |
| 業務ルールが複雑 | Clean Architecture |
| 複数領域で言葉の意味が違う | DDD / Bounded Context |
| 独立デプロイや独立スケールが必要 | サービス分割を検討 |

サービス分割は最後の手段として考えます。分割すると、通信失敗、分散トランザクション、監視、認証、バージョニング、ローカル開発が難しくなります。

まずは単一デプロイ単位の中で責務と依存方向を整え、それでも独立性が必要になった境界を分ける方が安全です。
