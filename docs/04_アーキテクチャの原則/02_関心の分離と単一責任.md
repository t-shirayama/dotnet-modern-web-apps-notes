# 関心の分離と単一責任

関心の分離は、アプリケーションを異なる理由で変更される部分に分ける考え方です。UI 表示、入力検証、業務判断、データ保存、外部 API 呼び出し、ログ出力は、それぞれ変更理由が違います。

単一責任の原則は、クラスやモジュールが「1 つの変更理由」を持つようにする考え方です。Controller が画面制御、業務判断、DB 更新、メール送信、ログ整形をすべて持つと、変更理由が増えすぎます。

| 責務 | 置き場所の例 |
| --- | --- |
| HTTP リクエストの受け取り | Controller / Endpoint |
| ユースケースの進行 | Application Service |
| 業務ルール | Domain model / Domain service |
| DB アクセス | Infrastructure / Repository |
| 外部サービス連携 | Infrastructure adapter |

分けすぎると理解が難しくなりますが、混ぜすぎると変更が怖くなります。最初は「業務判断」と「インフラ処理」を分けるだけでも効果が大きいです。
